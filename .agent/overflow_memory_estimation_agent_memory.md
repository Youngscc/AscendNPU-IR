# PlanMemory Overflow Memory Estimation Agent Memory

This file is an agent-facing memory document for the AscendNPU-IR repository.
It is intentionally kept outside `docs/` so it does not affect the Sphinx
documentation build.

For the modular memory system index, start from `.agent/README.md`.
This file is the full long-form snapshot.

## 1. Mission

The central task is to predict local memory usage before the real compiler
pipeline reaches PlanMemory overflow.

The user wants a precise and lightweight model that, given the current compile
options or compile config, can estimate the memory that will be required later
in the compilation process. The goal is to avoid reaching a real overflow in a
full compile environment, because overflow currently triggers fallback or retry
behavior.

The important target is not runtime memory on an Ascend device. The target is
the static memory planned by BiShengIR before downstream `hivmc` or CANN work.

## 2. Current Understanding

The overflow of interest is generated by BiShengIR's `hivm-plan-memory` pass.
It does not require CANN or a physical Ascend card to reproduce if the input IR
has already reached the HIVM/memref level.

The relevant failure format is:

```text
ub overflow, requires <N> bits while <M> bits available!
```

The diagnostic is emitted in:

```text
bishengir/lib/Dialect/HIVM/Transforms/PlanMemory.cpp
```

The message mentions common causes:

```text
tiling basic block is too large
block number is more than expected due to multi-buffer
some ops need extra local buffer
```

This matches the user's suspected scenario: increasing tile size can increase
intermediate local buffers and cause UB/CBUF/L0C overflow.

## 3. Non-Goals

Do not focus on these unless the user explicitly asks:

- Running kernels on a real Ascend device.
- Installing or debugging CANN.
- Making full `check-bishengir` pass on macOS.
- Fixing unrelated documentation warnings.
- Changing existing docs under `docs/`.

The missing `hivmc` problem is real for full compilation, but it is downstream
of the PlanMemory problem. It should not block this memory estimation task.

## 4. Important Vocabulary

IR stages:

- `ttir`: earlier tensor-level Triton-style IR. Usually all or mostly tensor.
- `ttadapter`: adapter/lowered IR. In the user's sample it contains both tensor
  and memref. It may be close to the point where this repository starts doing
  HIVM lowering and memory planning.
- HIVM memref IR: the useful level for PlanMemory. It contains memrefs with
  address spaces such as UB, CBUF/L1, and L0C/CC.

Memory spaces:

- `#hivm.address_space<ub>` maps to UB.
- `#hivm.address_space<cbuf>` maps to L1/CBUF.
- `#hivm.address_space<cc>` maps to L0C.
- `#hivm.address_space<gm>` is global memory and is not the local overflow
  target for the current task.

Plan modes:

- `local-mem-plan`: plans local `memref.alloc` buffers in UB/L1/L0C.
- `global-work-space-plan`: plans `memref_ext.alloc_workspace`.

For the user's tile-size overflow task, `local-mem-plan` is the main focus.

## 7. Existing Reproduction Command

This repository already has PlanMemory overflow examples.

Run from repository root:

```bash
build/bin/bishengir-opt \
  bishengir/test/Dialect/HIVM/plan-memory.mlir \
  --split-input-file \
  --hivm-plan-memory='mem-plan-mode=local-mem-plan'
```

Expected behavior: several intentionally bad splits fail with diagnostics like:

```text
ub overflow, requires 2560000 bits while 1572864 bits available
```

For a custom HIVM-level input:

```bash
build/bin/bishengir-opt input.ttadapter \
  --hivm-plan-memory='mem-plan-mode=local-mem-plan enable-memory-display=true' \
  --mlir-print-ir-after-failure
```

Notes:

- This only works if the input is already valid for `hivm-plan-memory`.
- If the input is too early, such as pure tensor TTIR, earlier lowering passes
  must run first.
- `enable-memory-display=true` writes `memory_info.json`,
  `memory_info_aic.json`, or `memory_info_aiv.json` in the current directory.
  Those files can be overwritten between runs.

## 8. Why Estimation Must Respect Compile Options

Memory usage is not determined only by tensor shape. It also depends on which
passes run before PlanMemory and which options are enabled.

Important options include:

| Option | Why it matters |
| --- | --- |
| `--enable-auto-multi-buffer` | Can multiply local buffer allocation. |
| `--limit-auto-multi-buffer-only-for-local-buffer` | Restricts where auto multi-buffer applies. |
| `--limit-auto-multi-buffer-of-local-buffer` | Controls local buffer multi-buffer strategy. |
| `--limit-auto-multi-buffer-buffer` | Controls cube/vector mixed multi-buffer strategy. |
| `--set-workspace-multibuffer` | Changes workspace multibuffer count. |
| `--tile-mix-cube-loop` | Changes loop tiling and can affect buffer lifetimes or sizes. |
| `--tile-mix-vector-loop` | Same, but vector side. |
| `--enable-preload` | Can introduce or expand preload-related buffers/lifetimes. |
| `--enable-code-motion` | Can change lifetimes and fallback behavior. |
| `--enable-memory-display` | Enables debug JSON after PlanMemory. |
| `--enable-hivm-global-workspace-reuse` | Affects workspace memory plan, not mainly local UB overflow. |
| `--disable-auto-cv-work-space-manage` | Can skip parts of workspace management and planning. |
| `--enable-triton-kernel-compile` | Enables overflow retry policies in `bishengir-compile`. |
| `--enable-tuning-mode` | Enables a different retry path for HFusion buffer tuning. |

Any exact estimator must take a snapshot of the same compile config that the
real compile would use.

## 9. Where The Exact Estimator Should Run

For exact local memory prediction, the estimator should run immediately before
the existing local `createPlanMemoryPass` in `HIVMPipelines.cpp`.

At that point, the IR has already had:

- extra buffers allocated,
- memory scope inferred,
- canonicalization applied,
- inline load copy processed,
- multi-buffer marks applied.

This is the right place because estimating earlier can be too optimistic.
Estimating after PlanMemory is too late because the overflow has already
happened.

For workspace estimation, add a similar hook before the earlier
`createPlanMemoryPass` configured with:

```text
memMode = GLOBAL_WORKSPACE_PLAN
```

But the user's current goal is mostly local memory overflow caused by tile size,
so prioritize local memory mode first.

## 10. Recommended Architecture

Use a two-level design:

### Level 1: Fast Approximate Precheck

Parse the current IR and collect obvious local `memref.alloc` buffers:

- shape,
- element type bit width,
- address space,
- static/dynamic status,
- multi-buffer annotations if already present.

Estimate:

```text
raw_bits = product(shape) * element_bit_width
aligned_bits = align_up(raw_bits, scope_align_bits)
```

Then compute a conservative upper bound per scope by summing aligned buffers,
including multi-buffer expansion.

This is fast but not fully precise because it does not model:

- liveness,
- in-place reuse,
- PlanMemory allocation order,
- aliasing,
- pass-created extra buffers,
- retry/fallback option changes.

### Level 2: Exact Dry-Run Estimator

This is the preferred design for the user's task.

Add a lightweight analysis pass or dry-run mode that reuses PlanMemory internals
without performing destructive IR rewriting and without emitting the overflow
diagnostic that triggers fallback.

The dry-run should reuse:

```text
MemLivenessAnalysis
MemPlan
BufferInfo
StorageEntry
buffer2Life
buffer2MultiNum
genKillMap
inplacePairList
```

The estimator should output structured data such as:

```json
{
  "function": "kernel_name",
  "mode": "local-mem-plan",
  "config": {
    "enable_auto_multi_buffer": true,
    "enable_code_motion": true,
    "tile_mix_cube_loop": 1,
    "tile_mix_vector_loop": 1
  },
  "scopes": [
    {
      "scope": "UB",
      "capacity_bits": 1572864,
      "required_bits": 1835008,
      "overflow": true,
      "buffers": []
    }
  ]
}
```

The output should avoid using the exact phrase `ub overflow` unless the intent is
to trigger the existing retry mechanism. Use a neutral phrase such as
`predicted_over_capacity`.

## 11. Why Not Reimplement Everything Outside The Compiler

A standalone script that parses MLIR text may be useful for a rough estimate,
but it will not be exact enough for this task.

Reasons:

- PlanMemory depends on operation order and nested regions.
- It computes liveness, gen/kill, aliases, and in-place reuse.
- It expands multi-buffer entries.
- It may retry planning with deterministic shuffle seeds.
- Earlier passes can create new temporary buffers.
- Compile options alter the pipeline.

For exact results, share the same C++ analysis data structures used by
PlanMemory.

## 13. How To Avoid Existing Rollback Behavior

`bishengir-compile` uses `RetriablePassManager`.

When `--enable-triton-kernel-compile` is enabled, overflow retry policies are
registered for:

- UB overflow,
- CBUF/L1 overflow,
- CC/L0C overflow.

Current overflow recovery roughly tries:

1. Disable `enable-code-motion`.
2. Disable `enable-auto-multi-buffer`.

That means a real overflow can cause the compiler to re-run with different
options. The user's goal is to avoid reaching that point.

Therefore the estimator should:

- run before the real PlanMemory pass,
- not emit the existing overflow diagnostic pattern,
- report predicted over-capacity in structured output,
- optionally allow a future policy to proactively choose safer tiling or config
  before the retry manager sees an actual PlanMemory failure.

## 14. Practical First Implementation Plan

Recommended next coding steps:

1. Add an internal memory estimate result type near PlanMemory or MemoryDisplay.
2. Refactor PlanMemory enough to expose a dry-run calculation path.
3. Add a pass such as `hivm-estimate-memory` or add a non-failing option to
   `hivm-plan-memory`, for example `estimate-only`.
4. Run the estimator at the same pipeline point as local PlanMemory.
5. Emit JSON to a user-selected path instead of always writing
   `./memory_info.json`.
6. Add lit tests using existing overflow examples in
   `bishengir/test/Dialect/HIVM/plan-memory.mlir`.
7. Add a macOS-friendly reproduction test that does not require `hivmc`.

Prefer a new pass over a separate external parser if precision is required.

## 16. Known Mac Environment Status

macOS can run the relevant static PlanMemory reproduction if the project has
already built `bishengir-opt`.

macOS does not need:

- CANN,
- `hivmc`,
- an Ascend card,

for this specific PlanMemory estimation task.

Full downstream tests and full `bishengir-compile` to binary may still fail on
macOS because `hivmc` is missing or because tests hard-code Linux `.so` paths.
That is separate from this task.
